1 Introduction to IoT ▾

What is IoT?

The Internet of Things (IoT) is a global network of physical objects ("things") connected to the internet that can collect, send, and receive data. Everyday items like toothbrushes, cars, and refrigerators can be made "smart" by embedding sensors and network connectivity into them.

Computing chips got small and cheap enough to put inside ordinary objects. This created an entire industry around connecting everyday things to the internet.

Three Core Elements of IoT

ElementWhat it doesExample
ThingsPhysical objects with sensors that gather, send, and receive dataSmart refrigerator, temperature sensor in a factory
ConnectivityNetwork technologies that let devices communicateFleet of delivery trucks using GPS to transmit location data
Sense-makingProcessing raw data into meaningful insights for decisionsFactory sensors predicting machine failure
IoT data is often time-series data (data points paired with timestamps).

IoT Applications

  • Connected cars: Dashcams, fuel monitoring, accident alerts, breakdown prediction
  • Connected homes: Smart plugs monitoring electricity, smart thermostats, lost-item finding
  • Smart cities: Air quality monitoring, smart street lighting, parking management
  • Smart buildings: Energy reduction, lower maintenance costs, workspace efficiency
  • Industrial IoT: Predictive maintenance, wearable safety devices, supply chain tracking

Benefits of IoT

  1. Improved quality of life — Automates daily tasks (coffee machine starts when alarm goes off)
  2. Boost innovation — Advanced analytics reveal new business opportunities
  3. Data-driven insights — Predict future outcomes from historical data
  4. Enhanced security — Continuous automated monitoring
  5. Customer satisfaction — Faster restocking, better services

A "thing" needs to be augmented with an embedded device that provides four capabilities:

Sensor Selection Process

Step 1 — Identify required sensing modalities

  • What context do you need? (e.g., "Is someone in the room?")
  • What sensor can detect that? (e.g., motion sensor for presence)
  • Where should it be deployed? (e.g., door sensor on the door)

Step 2 — Choose the best sensor

  • Check specifications (range, accuracy)
  • Decide: periodic readings (every X seconds) vs event-driven (only when something happens)

Step 3 — Other considerations

  • Power source (battery vs mains)
  • Size, weight, weather resistance
  • Budget
  • Compliance to health, safety standards
  • Interoperability with existing systems
  • Delivery lead time
  • Ease of mounting and unmounting
  • Maintenance requirements
  • End users' perceptions
  • Customized vs Commercial Off The Shelf (COTS)
  • Product reviews

Beacons

A beacon is a low-cost, low-powered device that periodically emits a signal (radio, audio, or light). Most common protocol: Bluetooth Low Energy (BLE), but anything that emits a signal periodically could be a beacon (e.g., Wi-Fi beacons).

How beacons work (RSSI)

  1. Beacon transmits a BLE radio signal with its ID, at a periodic interval, at a fixed transmit power
  2. A receiver within range using the same BLE protocol can decode the signal
  3. The further the receiver is from the beacon, the smaller the Received Signal Strength Indicator (RSSI)
  4. A receiver out of beacon range will not receive the signal at all

Key beacon uses

  • Geofencing: Define a virtual region; detect when devices enter/exit
  • Ranging/Proximity: Determine approximate distance
  • Indoor location: Useful where GPS doesn't work well (malls, airports)

3 Connectivity — TCP/IP & HTTP REST ▾

TCP/IP Model (5 Layers)

The TCP/IP model describes how data travels between computers. Think of it as a stack of layers, each with a specific job:

LayerResponsibilityExample Protocols
ApplicationServices for apps (web, email)HTTP, DNS, MQTT
TransportDelivers data to correct processesTCP, UDP, QUIC
NetworkAssigns addresses, routes across networksIP (Internet Protocol)
LinkReliable transfer across one single linkWi-Fi, Ethernet
PhysicalRaw signal transmissionRadio waves, cables

Key rules

  • Each layer can only talk to the layer directly above and below it
  • Applications don't need to care about which link technology is used
  • Packetization: Data is split into small packets, each with headers (metadata) and payload (actual content)
  • IP-based: Any device using IP can communicate with other IP-based devices

HTTP REST

HTTP is an application-layer protocol for transferring data. It is stateless: each request is independent.

REST is an architectural style using HTTP methods:

MethodPurposeExample
GETRetrieve dataGet temperature reading
POSTCreate new dataRegister a new sensor
PUTUpdate existing dataUpdate device settings
DELETERemove dataRemove a device

HTTP Interaction Flow

  1. Client sends request (method + URL + data)
  2. Server processes request
  3. Server sends response (status code + data)
  4. Client processes response
4 Connectivity — MQTT ▾

What is MQTT?

MQTT is a lightweight, publish-subscribe messaging protocol designed for constrained devices on unreliable networks. Perfect for IoT.

  • Application-layer protocol, runs on top of TCP
  • Small headers, compact binary format = minimal network usage
  • Maintains persistent connections = less power needed
  • MQTT is NOT an acronym anymore

Publish/Subscribe Pattern (Pub/Sub)

Unlike client-server, pub/sub uses a broker as a middleman:

Publisher ---> BROKER ---> Subscriber 1
                      ---> Subscriber 2
                      ---> Subscriber 3

Three types of decoupling

  1. Space decoupling: Publisher and subscriber don't need to know each other's IP/port
  2. Time decoupling: They don't need to be online at the same time
  3. Synchronization decoupling: Publisher doesn't wait for subscriber to receive

Roles

MQTT Client

  • Any device running an MQTT library connected to a broker
  • Can be a publisher, subscriber, or both

MQTT Broker

  • Central hub managing all messages
  • Receives, filters, and distributes messages to subscribers
  • Handles authentication, authorization, session data
  • Single point of failure — must be highly scalable and failure-resistant

Operations

OperationWhat happens
ConnectClient sends CONNECT, broker responds with CONNACK + status
PublishClient sends a message with a topic; broker forwards to subscribers
SubscribeClient sends SUBSCRIBE for topics; broker confirms with SUBACK
UnsubscribeClient sends UNSUBSCRIBE; broker confirms with UNSUBACK
DisconnectEither side sends disconnect message

Key points

  • Each MQTT client has a unique Client ID
  • Messages are data-agnostic (binary, text, XML, JSON — anything)
  • New subscribers don't get past messages unless Retained Messages are enabled
  • Publishers can set a message expiry interval

MQTT Topics

Topics are strings used by the broker to filter messages. They have a hierarchical structure separated by /:

campus/soss/level3/room321
campus/soss/level4/room482
campus/lkcsb/level2/room213

No need to pre-create topics — the broker accepts any valid topic.

Wildcards (Subscribe only)

WildcardSymbolWhat it doesExample
Single-level+Matches one levelcampus/soss/+/lift_lobby/light
Multi-level#Matches all remaining levels (must be last)campus/soss/#
Wildcards can only be used for subscribing, NOT for publishing.

Topics Best Practices

  • No leading / (creates unnecessary blank level)
  • No spaces (causes bugs)
  • Keep topics short (included in every message)
  • Use only ASCII characters
  • Embed Client ID in topic for authorization
  • Don't subscribe to # (receives ALL messages — bandwidth killer)
  • Design for extensibility: Topics don't need to be preallocated. Think about how topics can be extended for new features. A smart home adding new sensor types should be able to add them without changing the whole hierarchy.
  • Use specific topics, not general ones: Instead of home/bedroom for all values, create home/bedroom/temperature, home/bedroom/brightness, home/bedroom/humidity.

Quality of Service (QoS)

The QoS level defines the guarantee of delivery for a message. There are 3 QoS levels:

QoS LevelNameGuaranteeMechanism
QoS 0At most onceNo guarantee ("fire and forget")No acknowledgment. Relies on TCP. Message not stored or retransmitted.
QoS 1At least onceDelivered at least onceSender stores message until PUBACK received. May result in duplicates.
QoS 2Exactly onceReceived exactly onceFour-part handshake: PUBLISH → PUBREC → PUBREL → PUBCOMP. Safest but slowest.

QoS Rules

  • Publisher → broker: always uses the publisher's chosen QoS level
  • Broker → subscriber: uses the lower of QoS(publisher → broker) and QoS(broker → subscriber)
  • Example: Subscriber QoS 0, Publisher QoS 2 → broker delivers at QoS 0
  • Example: Subscriber QoS 2, Publisher QoS 1 → broker delivers at QoS 1

Message Filtering

The broker filters messages so each subscriber only receives messages of interest:

  • Topic-based filtering: Subscriber subscribes to topics of interest. Broker ensures subscriber gets all messages published to those topics. Primary method used in MQTT.
  • Content-based filtering: Subscriber filters based on message content using a filter language. Downside: content must be known beforehand and cannot be encrypted.
  • Type-based filtering: Subscriber filters by message type (e.g., all messages of type Exception).

MQTT Communication Patterns

Point-to-point

  • Two devices use a single MQTT topic as the communication channel
  • Also works one-to-many: publisher sends different messages to multiple subscribers using one unique topic per subscriber
  • Example: Room occupancy sensor publishes to sensor/occupancy/room7; a digital display subscribes to the same topic

Broadcast

  • One-to-many: the same message is sent to a large fleet of devices
  • Multiple devices subscribe to the same topic; sender publishes one message
  • Example: Weather station broadcasts to dt/us/NY/weather; all delivery vehicles in NY subscribe

Fan-in

  • Many-to-one (reverse of broadcast): multiple devices publish, one single subscriber aggregates
  • Subscriber may use wildcards to aggregate similar but unique topics
  • Example: Air quality sensors publish to environment/building/lax002/airqual/zone1, .../zone2; a component subscribes to environment/building/lax002/airqual/+

Retained Messages

  • An MQTT message includes a retained flag (boolean)
  • A retained message is stored by the broker as the last retained message for that topic, along with its QoS
  • Every topic can only have one retained message — a newer one replaces the older
  • Each new subscriber to a topic receives the retained message immediately after subscribing
  • However, they will NOT receive older non-retained published messages
  • Like joining a Telegram group with "Chat History" set to "Hidden" — you only get the last pinned message
  • Helps new subscribers get the last known status without waiting for the next publish

Last Will and Testament

  • When disconnecting, a client should send a DISCONNECT message. If it doesn't and the TCP connection drops, this is an ungraceful disconnection
  • A Last Will message notifies other clients about an ungraceful disconnection
  • Each client specifies its Last Will message (topic + payload) when connecting to the broker
  • The broker stores it and only sends it to subscribers of the Last Will topic when the client disconnects ungracefully
  • If the client disconnects gracefully with DISCONNECT, the broker discards the Last Will message

Persistent Sessions

  • In a non-persistent session, a disconnected client must reconnect and re-subscribe to all topics
  • A client can request a persistent session when connecting to avoid re-subscribing
  • The broker saves (identified by Client ID):
    • All the client's subscriptions
    • All messages in QoS 1 or 2 flow that were interrupted
    • All new QoS 1 or 2 messages missed while offline
    • All QoS 2 messages not yet completely acknowledged
  • A session expiry interval (in seconds) can be set. After disconnect, if the client hasn't reconnected within this period, the session is deleted
  • The client also stores: all QoS 1/2 messages not yet confirmed by broker, and all QoS 2 messages not yet completely acknowledged
  • When the client reconnects, stored information is available immediately

User Properties

  • Key-value string pairs appended to most types of MQTT messages (similar to HTTP headers)
  • Makes the MQTT protocol extensible for application-specific needs
  • Example: Publishers add schema:JSON or schema:XML1.1 so subscribers know the data format
  • Example: Publisher adds region:canada as a user property for traceability in complex IoT deployments

Shared Subscriptions

  • In standard MQTT, each subscriber receives a copy of every message on a topic
  • In a shared subscription, clients in the same group receive messages in an alternating (round-robin) fashion
  • Useful for client load balancing when a topic has a high message rate
  • Topic structure: $share/GROUPID/TOPIC
  • Three parts: $share (static identifier), group identifier, actual topic (may include wildcards)
  • Each subscription group acts as a virtual client — the broker selects one subscriber from the group to deliver each message

Alert Fatigue

When too many alerts overwhelm people, causing them to miss, ignore, or delay responses.

How to combat it

  • Set intelligent thresholds (not every alert needs immediate attention)
  • Find the right balance (too few = missed incidents; too many = alert fatigue)
  • Tiered priorities (red lights + voice for warnings; amber text for advisories)
  • Make alerts actionable (clear instructions on what to do)

Design Principles (Material Design)

  • Accurate: No distortion of data
  • Helpful: Provide context and useful insights
  • Scalable: Work across different screen sizes

Chart categories by purpose

  • Change over time: Trends and temporal comparisons
  • Category comparison: Compare between distinct groups
  • Ranking: Show items in ordered position
  • Part-to-whole: Show how parts sum to a total (stacked bar, pie, donut, stacked area, treemap, sunburst)
  • Correlation: Show correlation between variables (scatterplot, bubble, column/line, heatmap)
  • Distribution: How often values occur (histogram, box plot, violin, density)
  • Flow: Movement of data between states (Sankey, Gantt, chord, network)
  • Relationship: How items relate (network, Venn diagram, chord, sunburst)

Visual Encoding

Visual encoding is translating data into visual form. Graphical attributes for both quantitative and qualitative data:

  • Shape: Different shapes represent categories (circles, squares, triangles). Precise shapes for close exploration; less detail for general trends.
  • Color: Four primary ways: (1) distinguishing categories, (2) representing quantity, (3) highlighting specific data (use sparingly), (4) expressing meaning (e.g., red = danger). Always reinforce color with other visual cues — don't use color alone.
  • Size, Area, Volume: Larger = greater value
  • Length: Bar length corresponds to value
  • Angle: Pie chart slices
  • Position, Direction, Density: Location on axes, flow direction, concentration of points
Multiple attributes: A bar's color could represent a category, while its length represents a value.

Style

  • Graphical elements: Shapes, lines, colors, images to represent data
  • Typography: Font selection, size, spacing for readability
  • Iconography: Symbols/icons to depict concepts, reducing text reliance
  • Axes and labels: Set scales and reference points
  • Legends and annotations: Additional context to clarify meanings

Accessibility (WCAG)

  • Alternative text: Helps screen readers understand non-text elements
  • Contrast and color accessibility: Meet WCAG contrast standards so color differences are distinguishable to users with visual impairments
6 IoT Architecture & Design ▾

General Principles

PrincipleWhat it meansWhy it matters
Decouple ingestion from processingSeparate data collection from data processingScale processing independently based on load
Design for offline behaviorDevices may go offline for long periodsMust be delay-tolerant; monitor for silent devices
Lean data at edge, enrich in cloudStore compact data on device, add detail in cloudSaves device storage and transmission energy
Handle personalizationEach deployment site has unique settingsNeed UI or phone app for device setup
Regular status checks (heartbeats)Devices should regularly report they're aliveDetect offline devices and take action
Security at all layersPhysical nature of IoT creates unique vulnerabilitiese.g., NFC relay attacks on cars

Operational Excellence

  • Plan device provisioning: Create device identities securely, assign unique certificates
  • Device bootstrapping: Devices must connect and update config during setup; bootstrapping server must support all firmware versions ever deployed
  • Document communication patterns: After deploying a few devices, extrapolate total expected traffic
  • Over-the-Air (OTA) updates: Continuously update firmware, track success/failure, allow rollback
  • Functional testing on physical assets: Rigorous testing before field deployment
  • Design for operations at scale: Plan logging, monitoring, fleet management

Security Principles

  • Manage security lifecycle holistically: From design phase to hardware retirement/destruction
  • Least privilege permissions: Restrict which topics each device can use; compromised device = limited damage
  • Zero Trust: Never automatically trust anything; verify everything before granting access

Reliability Principles

  • Simulate at production scale: Test at 10% → 25% → 50% → 100% traffic
  • Buffer with queues: Incoming rate may exceed processing rate; queues prevent data loss
  • Design for failure: Retry logic, firmware rollback, protocol failover, redundant sensors, factory reset

Performance Efficiency

  • Use managed services (databases, compute, storage)
  • Decouple ingestion and processing for scalable data handling
  • Use event-driven architectures

Cost Optimization

  • Manage manufacturing tradeoffs: Cheapest vendor may cost more long-term
  • Avoid unnecessary data: Stop collecting unused data; adjust granularity and retention
  • Process at the edge: Upload only relevant insights to the cloud

Design Patterns

Command Pattern

Problem: How to reliably ask devices to perform actions across unreliable networks?

Solution: No action is deemed successful unless acknowledged as successful.

  1. Device configures itself to communicate via a protocol endpoint
  2. A component publishes a Command message to the device
  3. Server sends the Command to the device
  4. Device performs the action and publishes a completion message

Device Bootstrap Pattern

Problem: A new device starts with zero privileges and needs to become fully operational.

Solution: Two-phase approach:

  1. Registration Authority: Device registers with shared credentials/token → gets validated → receives short-lived credentials
  2. Privilege Escalator: Device shares more info → if authorized, receives long-lived credentials (certificates)
  3. Device can now subscribe/publish to its messaging topics

Device State Replica Pattern

Problem: Need to track and sync device states even with intermittent connectivity.

Solution: Maintain three things:

  • Current state: What the device reports
  • Desired state: What the system wants
  • Delta: The difference between current and desired

Component-to-Device flow (system wants to change device):

  1. Device reports initial state to state/deviceID/update
  2. Replica stores it
  3. Device subscribes to state/deviceID/update/delta
  4. Component publishes desired state
  5. Replica sends delta to device
  6. Device performs changes
  7. Device reports new state

Device-to-Component flow (device changes its own state):

  1. Device reports initial state
  2. Component subscribes to delta topic
  3. Device detects state change, reports new state
  4. Replica records and publishes delta
  5. Component receives the change

Gateway Pattern

Problem: Some IoT devices don't have direct internet access.

Solution: A gateway relays messages between devices and the cloud server.

  • Up gateway: Device → Gateway → Server (same topic, mirrored upward)
  • Down gateway: Server → Gateway → Device (same topic, mirrored downward)

Telemetry Pattern

Problem: IoT devices need to reliably sense and transmit data from their environment across intermittent networks.

Solution: Use the Telemetry pattern:

  1. Device obtains a sensor measurement
  2. Device publishes a message to telemetry/deviceID containing the measurement
  3. Server applies rules for fine-grained routing based on the message data
  4. Routing sends the message to other components of the solution

Telemetry Archiving Pattern

Problem: IoT solutions need real-time, batch-mode, and predictive analysis. Historical data must be preserved for future analysis.

Solution: Store raw unprocessed sensor data with timestamps supporting in-order replay:

  1. Device publishes measurement to telemetry/deviceID
  2. Server rules fan out messages to a processing path and a raw storage path
  3. Processing path computes results needed by other components
  4. Raw storage path saves original messages supporting in-order replay
  5. A future component can read raw messages from a point in time and replay them to telemetry/deviceID/replay

Software Update Pattern

Problem: IoT devices need software updates with minimal human intervention, at scale.

Solution: Combine the Command and Device State Replica patterns:

  1. Device subscribes to state/deviceID/update/delta
  2. Application places new software in accessible storage
  3. Application publishes desired state (with new software URL) to state/deviceID/update
  4. Device State Replica records desired state, publishes delta to device
  5. Device receives delta (the "software update" command) with the new software URL
  6. Device downloads and applies the new software
  7. Device publishes acknowledgement with new software version to state/deviceID/update
  8. Replica publishes to state/deviceID/update/accepted — update complete

MVC Device Software Design Pattern

Problem: IoT devices have common concerns: reporting state, processing control messages, and presenting status. The software should be reusable across platforms.

Solution: Apply the Model-View-Controller pattern:

ComponentRoleRules
ModelRepresents overall device state (signals, metrics, peripherals)Only component that reads from hardware via HAL. Should not write to device.
ViewHandles presentation of data from the ModelMultiple views possible (LEDs, JSON for network). Reads from Model, never from hardware directly.
ControllerTakes actions to change device state/configurationOnly component that writes to the device. View must use Controller for local actions.
  • Observer pattern: Hardware readings and network commands are observable data. View and Controller are observers that subscribe to changes.
  • Hardware Abstraction Layer (HAL): Abstracts hardware into API calls (e.g., readAccelerometer()). If hardware changes, only HAL needs updating — Model doesn't need to know.
7 Connectivity — LoRaWAN ▾

Radio Frequency Basics

  • Operating frequency: Two devices must use the same radio frequency to communicate
  • Signal bandwidth: Difference between max and min frequencies (Hz). NOT the same as data rate.

Key relationships

  • Higher bandwidth → Higher data rate (wider road = more cars)
  • Higher frequency → Lower range (shorter wavelengths = more susceptible to obstacles)

LoRa (Physical Layer)

LoRa = Long Range. A physical layer modulation technique using Chirp Spread Spectrum technology.

  • A chirp = a signal where frequency changes over time
  • Up-chirp: frequency increases. Down-chirp: frequency decreases.

Spreading Factor: How fast the frequency changes

  • High SF: Frequency changes slowly → longer range, lower data rate, higher energy, more airtime, more resilient to interference
  • Low SF: Frequency changes quickly → shorter range, higher data rate, lower energy, less airtime
  • Due to spread spectrum technology, communications with different data rates do not interfere with each other

Duty Cycles

  • Airtime: How long a device spends transmitting
  • Duty cycle: Fraction of time a device transmits (e.g., transmit 1s every 100s = 1%)
  • Governments regulate duty cycles to reduce interference. Typically 1% for LoRa.

LoRaWAN (Network Protocol)

LoRaWAN sits on top of LoRa. It's a Low Power, Wide Area networking protocol.

SpecValue
FrequencySub-GHz: 433 MHz, 868 MHz, 920 MHz
Data rateUp to 50 kbps
Max payload~200 bytes (as low as 11 bytes in some regions)
Encryption128-bit Network Session Key + 128-bit Application Session Key

Adaptive Data Rate: Network server optimizes spreading factor, bandwidth, and power per device.

Two encryption layers enable multi-tenanted shared networks without the network operator having visibility of user data.

End-device operating pattern

Devices follow a general cycle: sleep → transmit → receive → sleep

  1. Sleep: Device neither sends nor receives
  2. Transmit: When it has data, wakes and sends
  3. Receive: Listens for incoming data (receive windows)
  4. Sleep: Goes back to sleep

LoRaWAN Topology: Star-of-Stars

End-Device ---radio---> Gateway ---internet---> Network Server
End-Device ---radio---> Gateway ---internet---> Network Server
  • Gateways relay between end-devices and the central network server
  • Single-hop from end-device to gateway (or double-hop with relay)

LoRaWAN Network Components

ComponentRole
End-DeviceSensor/actuator wirelessly connected to LoRaWAN
Radio GatewayForwards all received packets to Network Server (doesn't process them)
Network ServerHandles MAC layer, data rate adaptation, routing, deduplication
Join ServerManages Over-the-Air device activation
Application ServerHandles application-layer data, provides service to end-users

Message Types

  • Uplink: End-Device → ALL gateways in range → Network Server. Deduplicated at Network Server.
  • Downlink: Network Server → ONE Gateway → End-Device. Buffered if device isn't ready.

LoRaWAN Operating Classes

ClassLatencyPowerHow it works
Class A (default)HighestLowestDevice sends whenever; opens 2 short receive windows after each uplink. Server can only send AFTER device sends.
Class B (beaconing)MediumMediumLike Class A + predictable ping slots. Beacons every 128s for time sync.
Class C (continuous)LowestHighestReceive window always open except when transmitting.

Class A detail

  1. Device transmits
  2. Waits ~1s, opens RX1 window (frequency/data rate based on uplink)
  3. Waits again, opens RX2 window (fixed frequency, configurable)

Class B detail

  • Gateways broadcast beacons every 128 seconds for time sync
  • 4096 ping slots between beacons
  • End-device wakes at predictable times to listen

Class C detail

  • Receive window continuously open, only closes during transmission
  • Best for devices with unlimited power

Activation Methods

  • ABP (Activation-by-Personalization): Device directly tied to a specific network. For known factory deployments.
  • OTA (Over-the-Air): Device performs Join procedure. For consumer devices bought off-the-shelf.

Relays

Extend range when end-devices can't reach gateways directly. A relay is a LoRaWAN end-device that forwards frames using Wake On Radio (WOR).

8 Connectivity — Thread ▾

What is Thread?

An open standard for reliable, low-power, wireless mesh networking. Designed for IP-based networking where every device gets an IP address.

  • IP-based networking: Every device gets an IP address, making it easy to communicate with any other IP device
  • Self-configuring and self-healing
  • Secure (all communications encrypted, devices authorized)
  • Scalable (hundreds of devices per mesh). Thread Domains feature enables multiple networks to work together seamlessly.
  • Resilient (no single point of failure by design)
  • Efficient (long battery life with duty cycling)
  • Uses spread spectrum technology at the physical layer for good immunity to interference

Node Roles

RoleDescription
RouterForwards packets, provides joining/security services, radio always on
End DeviceCommunicates with a single Router (Parent), doesn't forward packets, can sleep
Parent-Child relationship: End Device attaches to exactly one Router. Router = Parent, End Device = Child.

Device Types

Radio always on, maintains routing table, subscribes to all-routers multicast.

May not be always on. No routing table, no all-routers multicast. Always sends data to parent router.

TypeDescription
Minimal End DeviceRadio always on, but no routing overhead
Sleepy End Device (SED)Radio off during idle, wakes periodically to poll parent

Upgrading/Downgrading

  • REED → Router: When it's the only node in range of a new End Device wanting to join
  • Router → End Device: When a router has no children

Partitions

  • A Thread network may split into partitions (groups that can't reach each other)
  • Each partition has its own Leader
  • Partitions automatically merge when connectivity is restored
  • All partitions share the same security credentials

Mesh Link Establishment (MLE)

Thread uses the MLE protocol to configure links and disseminate network information:

  • Discover links to neighboring devices
  • Determine the quality of links to neighboring devices
  • Establish links to neighboring devices
  • Negotiate link parameters with peers

MLE disseminates: Leader data, Network data, Route propagation data.

Network Discovery and Formation

Network identifiers

  • 2-byte PAN ID
  • 8-byte Extended PAN ID (XPAN ID)
  • Human-readable Network Name

Active scan process

  1. Device broadcasts Beacon Request on a channel
  2. Routers/REEDs in range respond with Beacon (PAN ID, XPAN ID, Network Name)
  3. Repeat for each channel
  4. Either join existing network or create new one

Creating a new network

  • Select least busy channel + unused PAN ID
  • Become a Router, elect self as Leader
  • Broadcast MLE Advertisements

Joining an existing network (MLE Attach)

  1. Child sends multicast Parent Request
  2. Routers/REEDs send Parent Responses
  3. Child picks a Parent, sends Child ID Request
  4. Parent confirms with Child ID Response

Connected Dominating Set (CDS)

Routers must form a CDS:

  • Any Router can reach any other Router through Router-only paths
  • Every End Device is directly connected to a Router

Thread adds routers for: coverage, path diversity, redundancy, more children. Removes routers when: max 32 exceeded, or space needed elsewhere.

Upgrading to a Router

After attaching as a Child (End Device), a device may elect to become a Router:

  1. Child sends an Address Solicit message to the Leader, asking for a Router ID
  2. If the Leader accepts, it responds with a Router ID
  3. Child upgrades itself to a Router
  4. New Router sends a multicast MLE Link Request to neighboring Routers
  5. Routers respond with Link Accept and Request messages
  6. New Router responds to each with a unicast Link Accept to establish bi-directional Router-Router links

What is Matter?

An interoperable standard for smart home devices, aiming to replace proprietary protocols. Built on IPv6. Builds upon years of experience with low power 802.15.4 networks (a global standard for low-rate wireless personal area networks) and Wi-Fi smart home devices.

  • Application-layer protocol in the TCP/IP model
  • Runs over Wi-Fi, Ethernet, and Thread
  • Strong cryptography with well-defined Device Types and data modeling
  • Supports bridging other protocols (Zigbee, Bluetooth Mesh, Z-Wave). A Bridge is a member device of both a Matter network and another bridged IoT technology — giving bridged devices access to Matter ecosystems while Matter leverages those technologies' large user bases.
  • Multi-admin: Multiple administrators can control devices
  • BLE used for initial commissioning

Matter Controller and Commissioning

The Matter Controller (e.g., phone, smart hub) pairs with and controls Matter devices:

  1. During commissioning, controller uses BLE to securely pass network credentials and provision the device into Thread or Wi-Fi
  2. Controller has the Commissioner role at this stage
  3. Once complete, device joins IPv6 network with all info needed to operate securely
  4. After joining, IPv6 communication is used for application messages (BLE no longer needed)

Matter Network Topology

  • Composed of Ethernet, Wi-Fi, and Thread devices within a local fabric
  • All communication over IPv6
  • Central hubs (Thread Border Router, Wi-Fi AP) connect different networks
  • Can work without internet (within firewalled network)

Device Types

A Device Type is an officially defined collection of mandatory and optional Clusters on one or more Endpoints. If a device implements a Device Type, it must implement all mandatory Clusters.

Each definition comprises: Device Type ID, Device Type revision, and one or more mandatory clusters and revisions. Device Types are specified in the Device Library (not the main Matter spec). Application Clusters are in the Application Cluster Library.

Defines how Nodes communicate. The Initiator starts interactions, the Target responds.

Groups and Paths

  • Groups: Address multiple devices simultaneously using the same Group ID (IPv6 Multicast)
  • Paths: Location of an Attribute/Event/Command in the hierarchy (e.g., Endpoint1.DoorLock.LockType)
  • "Popcorn effect": When sending individual commands to each device causes visible sequential delays. Groups solve this.

Timed vs Untimed Transactions

Timed Transactions set a maximum timeout to prevent Intercept Attacks:

  1. Alice sends message to Bob
  2. Eve intercepts it (e.g., radio jamming)
  3. Eve collects a second message
  4. Eve replays the first message to Bob
  5. Eve holds the second for later replay (security breach: "open lock")

Solution: Timed transactions expire, so Eve's replayed message is rejected.

Only use timed transactions for critical operations (locks, garage doors) — they add complexity.

Fabric

  • A Fabric = set of devices sharing the same security domain
  • Commissioning = assigning Fabric credentials to a new Node
  • Multi-Admin = Nodes can be on multiple Fabrics (e.g., manufacturer's + cloud ecosystem's)
  • Cluster Attributes/Events/Actions are shared between Fabrics

Matter + Thread Integration

  • Matter uses IPv6 for all operational communications
  • Thread Border Router bridges Thread and Wi-Fi/Ethernet networks
  • Border Router is a Stub Router (connects stub/last-mile network to regular network)
  • Responsibilities: auto-configure IPv6, publish multicast DNS for Thread Nodes, forward only relevant packets to Thread (Thread bandwidth is orders of magnitude less than Wi-Fi — bridging all Wi-Fi multicast would flood it)
  • Border Router connects the stub network (Thread) to the Adjacent Infrastructure Network (local Wi-Fi or Ethernet)

Low Power and Proxy Architecture

  • Some Matter Nodes are wired (radios always on). Others are battery-powered, using Thread.
  • The proxy architecture, along with Thread Sleepy End Devices, allows full-powered Nodes to provide both network-level and application-level functionality that insulates child Nodes from energy-intensive transactions.
